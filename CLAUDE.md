# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

MarketPulse is a real-time financial dashboard built to explore high-frequency frontend architecture: a Module Federation microfrontend monorepo (Lit + Lit Signals) fed by a WebSocket market simulator.

## Commands

```bash
npm install
npm run build:packages   # MUST run before typecheck/build:apps (apps import @market-pulse/* from each package's dist/)
npm run dev              # starts the simulator (:4000) + all 6 rspack dev servers (shell :3000, remotes :3001–:3005)

npm test                 # Vitest run (all tests)
npm run test:watch       # Vitest watch mode
npm run coverage         # Vitest + v8 coverage; FAILS if any file is < 80% (per-file threshold)
npm run typecheck        # tsc --noEmit across all apps + server (needs build:packages first)
npm run build            # full build, ordered: packages -> apps -> server
```

Run a single test file or case:
```bash
npx vitest run test/state-assets.test.ts
npx vitest run -t "dedups price per symbol"
```

Tests resolve `@market-pulse/*` from **source** (via Vitest aliases), so they do *not* require `build:packages`. Typecheck and the rspack app builds resolve from `dist/`, so they do. Build order is load-bearing — `build` and `build:packages` list packages in dependency order (contracts → utils → state → market-client → ui); the bare `tsc`/rspack steps assume upstream `dist/` already exists.

## Architecture

**Microfrontend topology (Module Federation).** `apps/shell` is the host; it lazy-loads five remotes (`market-overview`, `watchlist`, `asset-details`, `trade-stream`, `alerts`) at runtime. Remote rspack configs are generated by `tools/rspack-remote.cjs`; the host config is `apps/shell/rspack.config.cjs`. **The shared cross-app state only works because `lit`, `@lit-labs/signals`, `signal-polyfill`, and every `@market-pulse/*` package are declared `singleton: true`.** If a shared dep loses its singleton flag (or versions diverge), each remote gets its own isolated signal store and the apps stop seeing each other's updates.

**Two-tier reactive state (`packages/state`).** This is the core design and spans several files:
- Tier 1 — per-symbol `Signal.State<Asset>` kept in a `Map` (`asset-store.ts`). A price tick writes only that one symbol's signal, so only components observing that symbol re-render.
- Tier 2 — derived `Signal.Computed` aggregates (`derived.ts`: `topGainers`, `marketBreadth`, `assetsByType`, …) and `watchlistAssets`, which read across the per-symbol signals.
- Components extend `SignalWatcher(LitElement)` from `@lit-labs/signals`; any signal read during `render()` auto-subscribes that component.

**⚠️ The `assetStore` Object.is invariant.** signal-polyfill compares with `Object.is`, so setting a signal to a value it already holds is a *silent no-op*. `setAssets` therefore publishes a **new** `Map` reference (`assetStore.set(new Map(_assetMap))`) when symbols are added — mutating the existing Map in place would never notify the Tier-2 aggregates and would freeze every overview/watchlist view. `test/asset-store.test.ts` pins this regression; preserve it.

**Data pipeline (server → UI).** `server/market-simulator` runs a tick loop (`engine.ts`) that emits batched `MarketEvent`s over WebSocket. `MarketClientManager` (`packages/market-client`) routes messages: snapshots → `setAssets`, scenario starts → a synthetic alert, and event batches → a **RAF batcher** (`packages/utils/src/scheduling.ts`) that coalesces to one flush per animation frame. On flush it dedups price updates per symbol (last wins) and ingests trades/alerts as one batch — collapsing thousands of events/sec into ~60 signal-write bursts/sec.

**Control seam, not globals.** Backend actions (scenario injection, stress test) go through `getMarketControl()` / `setMarketControl()` in `@market-pulse/state`. The shell registers the REST client into this seam during `MarketClientManager.initialize()`, so remotes drive the backend without a `window` global or a direct dependency on `market-client`.

**Shared cadence profiles.** The stress-test toggle and the server both reference `NORMAL_PROFILE` / `STRESS_PROFILE` / `ENGINE_LIMITS` from `@market-pulse/contracts`, so the two ends can't silently disagree on tick rate/batch size; the server clamps incoming config to `ENGINE_LIMITS`.

## Testing notes (Vitest + happy-dom)

Config is `vitest.config.ts`: happy-dom environment (Lit components render in-process), coverage aliased to `src` for source-mapped reports, and a `.js`→`.ts` resolver plugin (source uses NodeNext-style explicit `.js` specifiers). Two non-obvious patterns are required and documented inline in the test files:

- **Component tests must side-effect-import the component *before* importing `@market-pulse/state`** (e.g. `import '../apps/alerts/src/alerts-panel';` as the first line). Otherwise `@customElement` registration is skipped and `createElement` yields an un-upgraded `HTMLElement` (no shadow root).
- **Mock `WebSocket` by assigning `globalThis.WebSocket` directly and restoring it** (see `test/helpers/fake-ws.ts` usage), not via `vi.stubGlobal` — churning a class-valued global stub with `unstubAllGlobals` throws "cannot be invoked without 'new'".
- After mutating a signal on a mounted component, await a macrotask then `updateComplete` (the `settle()` helper) — SignalWatcher schedules its update asynchronously.

## Conventions

- TypeScript is configured for legacy Lit decorators in `tsconfig.base.json` (`experimentalDecorators: true`, `useDefineForClassFields: false`). Packages build with `tsc` → `dist/`; apps bundle with Rspack + `ts-loader`.
- Custom elements are `mp-*`; the design system in `packages/ui` is intentionally minimal (`mp-mini-chart`, `mp-price-chart`) — apps style their own dense tables inline rather than via shared leaf components.
- CI (`.github/workflows/ci.yml`) gates every push on build → typecheck → coverage (per-file ≥80%) → production build.
